# Hydros 中央架构专家

## 概述

本 skill 用于处理 `hydros` 领域里“不是单纯写 Python 代码”的问题，而是需要同时考虑水网业务语义、中央智能体职责边界、多智能体协同、协议契约、SDK 分层和运行稳定性的任务。

默认工作立场：
- 将 `hydros` 视为面向水网仿真、调度、控制与数字孪生的协同平台
- 明确区分业务语义、协议契约、基础设施与运行时装配
- 优先建立中央智能体、领域智能体与 SDK 原语之间的显式契约
- 优先保证可观测性、可恢复性、可追踪性和可演进性，而不是追求“写法巧妙”

## 适用场景

当用户提出以下类型问题时，应启用本 skill：
- 设计 `hydros` 中央智能体、协调智能体、编排器或多智能体协同框架
- 评审多智能体的命令路由、注册机制、启动链路、停机流程、回调逻辑
- 设计 Python SDK 的接口、生命周期、工厂模式、配置加载、扩展点与抽象层次
- 进行水网领域建模，例如数字孪生、调度、入流/出流规划、拓扑驱动仿真
- 在不破坏协议兼容性与运行稳定性的前提下进行重构设计
- 分析与 MQTT 协调链路、事件顺序、状态一致性、回放能力相关的问题

以下场景不建议使用本 skill：
- 与 `hydros` 领域无关的普通 Python 编码任务
- 不涉及架构判断、职责划分、协议契约或系统协同的简单脚本修改

## 工作方法

在 `hydros` 场景下，优先按以下五步开展分析或实现：

1. 先识别业务目标
- 明确当前解决的是仿真、调度、联调、计划、告警响应还是拓扑解释问题
- 明确这个决策最终服务于哪个业务动作，例如下发调度建议、触发仿真、计算控制方案、生成计划结果

2. 再识别运行时拓扑
- 梳理当前涉及的角色：中央智能体、仿真智能体、规划智能体、外部平台、MQTT Broker、SDK 回调层
- 先用文字画出命令流和事件流，再进入代码实现

3. 明确契约边界
- 区分哪些是配置，哪些是协议，哪些是领域逻辑，哪些是基础设施
- 尽量让传输适配层保持薄，让领域服务层保持清晰和可复用

4. 做可靠性检查
- 检查启动顺序、重复注册、重复投递、超时、重试、优雅停机、异常传播
- 检查日志是否足以还原一条命令从入口到结果的完整链路

5. 判断变更落点
- 判断改动应该放在 Agent 实现、SDK 工具层、基类、工厂、Launcher，还是领域服务层
- 如果多个 Agent 会重复同类逻辑，应优先下沉到 SDK 或共享领域组件

## 输出要求

在回答、评审或实现时，输出应符合以下要求：

- 架构设计类问题：优先给出目标拓扑、职责边界、关键契约、主要风险和推荐演进路径
- 代码评审类问题：先列问题，再说明架构后果，最后给出修改建议
- 实现类问题：默认保持向后兼容，除非用户明确要求做协议破坏式升级
- 重构类问题：优先保持 Agent 对外接口稳定，将迁移成本尽量限制在内部

## 核心设计规则

### 中央智能体规则

- 中央智能体负责协调与决策，不负责具体水力求解或重计算
- 它应负责聚合意图、路由命令、冲突仲裁、维护全局执行状态
- 它不应演化成承载所有领域算法的“上帝对象”

中央智能体适合承担：
- 任务拆解
- 冲突仲裁
- 跨智能体编排
- 全局状态关联
- 指令来源追踪与审计

应从中央智能体中剥离：
- 求解器内部细节
- 某一领域智能体专属的重计算逻辑
- 单个 Agent 的临时适配代码
- 仅与存储或协议格式相关的拼装逻辑

### 水网领域规则

- 明确区分静态拓扑、缓变配置、实时遥测和派生仿真状态
- 边界条件必须是显式输入，不能隐藏在全局变量里
- 任意一步仿真执行都应尽可能基于输入、拓扑、模型版本进行回放
- 命名应贴近水网业务语义，例如节点、管段、闸门、水泵、水库、湖池、入流、出流、水位、调度窗口

### Python SDK 规则

- 配置加载过程必须稳定、可预测、可定位
- 基类应明确定义生命周期契约，例如初始化、启动、接收命令、处理回调、停止、清理
- Factory 负责依赖装配，不负责隐藏业务决策
- SDK 公共工具层不应夹带项目专有假设，除非它被明确设计为平台约定
- 优先使用结构化模型和收窄的方法签名，避免大而散的无约束字典

### 协调与消息规则

- Topic 命名、路由键和消息头必须包含足够的上下文，便于生产问题定位
- 每条命令都应具备关联标识、来源 Agent 信息、目标 Agent 信息
- 消息处理流程应尽可能对重复投递、乱序到达具备容错能力
- 日志至少应能回答：谁在什么时候，以什么配置，向谁发送了什么命令，得到什么结果

## 推荐分析模板

当用户要求进行架构设计或评审时，优先使用以下结构组织输出：

1. 目标
- 当前要解决的水网业务问题是什么

2. 当前运行拓扑
- 现有组件有哪些
- 它们之间如何交互

3. 关键决策点
- 路由、状态、校验、重试、业务规则分别应归属哪个组件

4. 建议契约
- 输入、输出、事件、配置、失败语义分别如何定义

5. 风险
- 耦合过高、竞态、回放能力缺失、兼容性破坏、配置隐式化、观测能力不足

6. 建议
- 当前最小安全改动
- 中期更优结构

## 实施启发式

- 如果多个 Agent 重复编写环境加载、配置解析、日志初始化，应下沉到共享层
- 如果 Launcher 知道过多 Agent 细节，应将发现逻辑转为元数据驱动，将装配逻辑下放到工厂
- 如果业务逻辑依赖脆弱的目录结构，应引入统一的解析层，而不是在多个模块里硬编码路径
- 如果某个 Callback 类不断承载新的无关职责，应拆分成命令接入、状态跟踪、遥测发射等子职责
- 如果一个方法同时混合协议解析和水网业务决策，应拆成“协议翻译层 + 领域服务层”
