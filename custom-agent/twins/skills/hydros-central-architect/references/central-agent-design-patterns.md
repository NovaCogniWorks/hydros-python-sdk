# 中央智能体设计模式

本参考文档用于指导 `hydros` 中央智能体的职责设计、内部结构划分和演进边界控制。

## 适用场景

当用户需要以下内容时，读取本文件：
- 设计中央智能体职责边界
- 设计跨 Agent 协同与任务编排机制
- 分析中央智能体是否过载
- 规划中央智能体的可演进结构

## 中央智能体的目标

中央智能体的核心目标不是替代所有专业 Agent，而是对全局业务流程进行统一协调。

它通常负责：
- 接收来自平台或上游系统的全局任务
- 将任务拆解成可执行的子任务
- 选择执行顺序、并发策略与回退策略
- 汇总多个 Agent 的结果
- 对冲突结果进行仲裁
- 输出最终可消费的决策结果或执行建议

## 推荐内部结构

建议将中央智能体拆分为以下子层：

### 1. Command Intake

职责：
- 接收外部命令
- 做基础校验
- 生成关联 ID
- 建立请求上下文

输出：
- 标准化任务上下文对象

### 2. Orchestration Planner

职责：
- 根据任务类型生成执行计划
- 决定调用哪些 Agent
- 决定执行顺序、并行度、超时和回退策略

输出：
- 可执行编排计划

### 3. Execution Coordinator

职责：
- 实际发起调用
- 追踪执行进度
- 处理成功、失败、超时和取消
- 做幂等控制和重复命令保护

输出：
- 分阶段执行结果

### 4. Result Arbiter

职责：
- 聚合多个 Agent 的结果
- 处理冲突、优先级和一致性问题
- 形成统一输出

输出：
- 最终业务结果

### 5. Audit And Telemetry

职责：
- 记录关键日志
- 输出指标
- 建立链路追踪信息
- 保留可回放元数据

## 典型设计模式

### 模式一：命令驱动编排

适合：
- 平台主动下发任务
- 中央智能体被动接收请求
- 执行链路明确

特点：
- 易审计
- 易追踪
- 便于做 SLA 管理

### 模式二：状态驱动编排

适合：
- 多源状态变化触发协同
- 需要根据系统状态选择 Agent
- 规则较多

特点：
- 更灵活
- 对状态一致性要求更高
- 更依赖事件语义设计

### 模式三：计划驱动编排

适合：
- 涉及时间窗、批量任务、预演和调度优化
- 需要中央智能体维护全局目标

特点：
- 强依赖计划模型
- 更适合调度和推演类业务

## 常见失控信号

如果出现以下现象，通常说明中央智能体职责已经失控：
- 中央智能体直接 import 多个专业 Agent 的内部求解逻辑
- 大量业务规则直接写在消息回调函数里
- 中央智能体同时负责编排、求解、格式转换和落库
- 新增一个 Agent 时必须频繁修改中央智能体多个核心模块
- 无法清晰回答“这个决策为什么属于中央智能体而不是专业 Agent”

## 收敛原则

- 中央智能体只保留全局视角下必须承担的职责
- 专业计算下沉到专业 Agent
- 协议拼装下沉到 SDK 或协议层
- 结果聚合与冲突仲裁尽量结构化
- 不要让中央智能体成为历史兼容逻辑的堆放点
