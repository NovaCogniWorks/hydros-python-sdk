# 架构评审清单

在评审 `hydros` 功能、Launcher、Agent 或 Python SDK 设计时，使用本清单。

## 业务契合度

- 当前水网业务目标是否清晰
- 关键决策归属是否清晰
- 静态拓扑、实时遥测、派生状态是否已分离
- 边界条件是否显式、可版本化、可追踪

## Agent 职责边界

- 每个 Agent 是否承担清晰且完整的领域能力
- 中央智能体是否在做协调，而不是替代专业智能体做计算
- 跨 Agent 依赖是否显式
- 一个 Agent 失败时，是否会污染无关流程的状态

## SDK 与扩展性

- 基类是否清晰定义生命周期钩子
- Factory 是否只做依赖装配，而不是偷偷注入业务策略
- 共享工具是否真的足够通用，适合下沉到 SDK
- 若未来出现第二个类似 Agent，这套抽象是否仍自然可用

## 协调与消息

- 消息类型与负载版本是否显式
- 每条命令链路是否具备关联 ID
- 对重试、重复投递和乱序消息是否具备安全性
- 运维人员是否可以通过日志追踪一条完整业务请求

## 配置治理

- 配置来源优先级是否明确
- 环境差异是否已从业务代码中隔离
- 启动时对关键缺失配置是否快速失败
- 路径与目录假设是否被集中治理而非散落硬编码

## 运行安全性

- 启动顺序与停机顺序是否可预测
- 局部注册失败是否可安全回滚或中止
- 异常日志是否包含远程排障所需上下文
- 超时、重连、重试策略是否有边界且可观测

## 重构安全性

- 对外协议是否保持兼容
- 若必须变更契约，是否存在过渡适配层或迁移路径
- 测试是否覆盖契约边界，而不只是内部实现
- 改动是否实质降低了耦合，而不是简单转移复杂度

## 建议输出格式

在评审响应中，优先使用以下结构：

1. 发现
- 严重级别、文件或模块位置、具体问题

2. 后果
- 该问题会导致什么运行风险、维护风险或架构退化

3. 建议
- 当前最小安全修复
- 如果允许继续演进，给出更优结构方案
