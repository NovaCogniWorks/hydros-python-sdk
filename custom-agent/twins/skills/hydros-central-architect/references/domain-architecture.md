# 领域架构参考

## 使用范围

当任务需要比主技能正文更强的领域架构视角时，读取本参考文档。

适用场景：
- 设计 `hydros` 总体架构
- 拆分中央智能体与领域智能体职责
- 分析水网业务对象、指令流与运行分层
- 设计中央协调、仿真、规划、控制等能力的边界

## 核心运行分层

### 1. 业务决策层

职责：
- 调度意图生成
- 执行策略选择
- 全局目标确定
- 跨智能体冲突仲裁
- 编排策略制定

典型组件：
- 中央智能体

这一层应以业务语义表达决策，而不是以传输语义表达决策。

典型例子：
- 在预报窗口内选择水库下泄策略
- 决定是否先运行数字孪生仿真再生成调度建议
- 决定是先做本体推理、先做计划推演，还是并行执行

### 2. 领域服务层

职责：
- 水力仿真流程
- 计划编制算法
- 状态派生
- 拓扑解释
- 领域校验

典型组件：
- Twins Agent
- 出流计划 Agent
- 中央调度 Agent

这一层应尽量可复用，不应过度依赖消息传输层细节。

### 3. 协调与协议层

职责：
- 命令封装
- 回调注册
- 关联 ID 管理
- 消息分发
- Broker 交互
- 生命周期信号处理

典型组件：
- Coordination Client
- Callback 适配层
- 协议模型

这一层必须稳定、显式、可审计。业务规则应调用它，而不是隐藏在它内部。

### 4. 平台工具层

职责：
- 配置加载
- 日志初始化
- 指标采集
- 重试工具
- 序列化工具
- Factory 组装模式

典型归属：
- Python SDK 共享模块

## 推荐职责划分

### 中央智能体

应该负责：
- 编排计划制定
- Agent 调用顺序控制
- 多结果冲突处理
- 命令来源与链路追踪
- 全局流程状态维护
- 面向运营者的决策摘要输出

不应该负责：
- 低层水力求解计算
- 某一专用 Agent 的领域算法细节
- 单个 Agent 的文件路径兼容处理
- 单纯的基础设施装配工作

### 专业领域智能体

应该负责：
- 专项领域计算
- 将业务请求映射为领域执行动作
- 校验自身输入
- 产出标准化领域结果

不应该负责：
- 跨领域冲突仲裁
- 与自身无关的全局流程状态维护
- 平台范围内的配置治理规则

## 稳定性设计模式

### 契约优先的消息设计

在修改处理逻辑前，应先定义清楚消息契约。

一条命令或事件至少应包含：
- 消息类型
- 关联 ID
- 来源 Agent 编码
- 目标 Agent 编码或广播范围
- 时间戳
- 负载版本
- 负载正文

### 可回放性

一次仿真或计划执行应尽可能可由以下信息重建：
- 拓扑版本
- 边界条件输入
- 模型/参数版本
- 命令元数据
- 执行时间点或时间窗

如果这些信息缺失，生产排障和结果追责会迅速失效。

### 幂等性

要默认接受以下情况可能带来的重复触发：
- 连接重建
- 超时后重试
- Broker 重投
- 人工回放

因此处理器要么幂等，要么能明确拒绝重复请求，并留下可追踪日志。

## 常见反模式

- 中央智能体直接 import 大量专业 Agent 内部实现
- 回调对象内混入大量业务规则
- 不带版本的自由字典负载
- 启动逻辑依赖脆弱目录结构
- 环境变量、配置文件、代码默认值之间的优先级不透明
- 日志里没有关联 ID
- 一个模块同时负责发现、配置、传输、装配和业务解析

## 推荐重构顺序

当 `hydros` 代码库已经变得难以维护时，建议按以下顺序重构：

1. 先把契约显式化
2. 再隔离启动与配置解析逻辑
3. 再拆分协议适配层和领域服务层
4. 再收缩中央智能体职责
5. 最后把重复装配逻辑收敛到 SDK 抽象层
