# Python SDK 架构演进任务拆解

## 1. 文档目的

本文将 Python SDK 的后续演进工作拆解为架构级任务清单，覆盖模块方向、对象层补齐、运行时增强、验证与阶段里程碑。

本文属于 `architect_agent` 的上游任务拆解文档，目标是把前面专题文档中的架构结论转成可排期、可交接、可验收的任务主线。

本文不直接替代：

- `engineering_agent/inputs/` 下的实施任务单
- 工程侧的文件级改造清单
- 执行侧的脚手架、demo 和交付落盘计划

## 2. 适用范围

适用范围：

- `hydros_agent_sdk` 的架构演进
- Python SDK 面向 DMPC 的对象层与协议层补齐
- 局部控制器、支持性 Agent 和多 Agent 宿主增强
- 与中央协调体系对接的工程实施规划

## 3. 总体实施策略

建议按四条主线推进，但以“对象和协议优先，运行时和业务后置”为原则。

四条主线：

- 主线 A：协议与领域对象补齐
- 主线 B：运行时和多 Agent 宿主增强
- 主线 C：局部 DMPC Agent 抽象与样例方向
- 主线 D：验证体系、样例工程与文档收敛

推荐顺序：

1. 先补对象模型和协议模型
2. 再增强状态管理与多 Agent 宿主
3. 再引入局部 DMPC Agent 抽象层
4. 再做中央对接与场景验证

## 4. 阶段拆解

### 阶段 1：协议与对象层补齐

目标：

- 不破坏现有 SDK 主链路
- 先让 Python SDK 具备表达 DMPC 语义的能力

建议交付方向：

- DMPC 对象模型模块
- DMPC 命令模块
- 配套的最小配置模型
- 序列化与反序列化验证

### 阶段 2：运行时增强

目标：

- 在现有 `BaseHydroAgent` 和 `MultiAgentCallback` 基础上补足控制区运行能力

建议交付方向：

- 控制区上下文容器
- 边界量缓存机制
- 多 Agent 路由增强
- 局部控制运行状态管理

### 阶段 3：局部控制器抽象和样例

目标：

- 建立 Python 侧局部 DMPC Agent 抽象层
- 完成单控制区试点样例

建议交付方向：

- `LocalDmpcAgent`
- 单区控制样例
- 局部求解输入输出与反馈样例

### 阶段 4：中央对接与场景验收

目标：

- 将 Python SDK 与中央协调输出正式打通
- 建立单区、双区和多 Agent 试点验收路径

建议交付方向：

- 中央目标下发协议适配
- 执行反馈回传协议适配
- 场景化验收清单与样例

## 5. 主线 A：协议与领域对象补齐

### A1. 新建 DMPC 对象模型模块

建议方向：

- 在 `hydros_agent_sdk/protocol/` 下新增独立 DMPC 对象模型模块
- 对控制区、耦合、协调输入输出、局部求解、执行反馈等对象进行结构化建模
- 保持字段命名与现有协议模型风格一致
- 避免把业务算法参数过度硬编码进基础对象

### A2. 新建 DMPC 命令模块

建议方向：

- 建立独立 DMPC 命令模块
- 保持与 `SimCommand` 体系兼容
- 使中央协调输入、局部求解输出、边界交换、执行反馈等对象可以标准化进入现有协议宿主

### A3. 扩展命令分发入口

建议方向：

- 将 DMPC 命令对象纳入统一 Envelope 分发入口
- 保持新旧命令兼容
- 为后续按模块拆分命令保留兼容导出层

### A4. 补齐 DMPC 事件对象

建议方向：

- 建立与命令分离的 DMPC 事件对象层
- 支撑局部控制器内部状态变化、边界更新和 fallback 触发等语义表达

## 6. 主线 B：运行时和宿主增强

### B1. 新增控制区上下文容器

建议方向：

- 在 `SimulationContext` 之下引入控制区上下文容器
- 保存控制区标识、受管对象、边界缓存、目标快照和反馈快照
- 不替代既有任务上下文，而是在其下形成子层次

### B2. 增强 `AgentStateManager`

建议方向：

- 增加 `context -> zone contexts` 映射
- 增加局部求解状态缓存、边界量缓存和 fallback 状态记录
- 保持既有任务状态管理不被破坏

### B3. 增强 `MultiAgentCallback`

建议方向：

- 增加 `context -> zone -> agent` 路由能力
- 支持 DMPC 目标定向派发和边界量交换
- 支持局部控制器与支持性 Agent 混合托管

### B4. 扩展 `MessageFilter`

建议方向：

- 为 DMPC 新命令提供过滤规则
- 明确区级路由命令、本地回环响应和远端局部求解报告的过滤口径

### B5. 扩展 `SimCoordinationClient`

建议方向：

- 注册 DMPC 新命令处理入口
- 增加更清晰的日志上下文标识
- 保持初始化、Tick 和时序更新主链路不受破坏

## 7. 主线 C：局部 DMPC Agent 抽象与样例方向

### C1. 新增局部 DMPC Agent 抽象基类

建议方向：

- 在 `BaseHydroAgent` 之上增加局部控制抽象层
- 定义目标接收、局部求解、反馈构建和边界更新等标准钩子
- 不在基类中绑定具体优化器实现

### C2. 新增边界感知 Agent 抽象

建议方向：

- 建立处理上下游边界量的抽象层
- 将边界缓存、同步策略和更新钩子沉淀为复用能力

### C3. 整理现有示例 Agent 角色

建议方向：

- 明确哪些 Agent 是抽象宿主，哪些是可直接复用示例
- 明确 `CentralSchedulingAgent` 是抽象中央宿主，不等价于完整中央实现
- 为仿真 Agent 增加与控制区和边界更新相关的扩展示例方向

### C4. 新增单控制区样例方向

建议方向：

- 提供单区局部控制器的最小闭环样例
- 配套环境配置、实例配置、示例 YAML 和说明文档

说明：

- 样例工程落地和对外交付整理由 `execution_agent` 负责
- 本文只定义样例应该验证的架构意图和能力边界

## 8. 主线 D：配置、验证、文档与样例工程

### D1. 扩展 YAML 配置模型

建议方向：

- 为 DMPC 配置增加扩展容器或显式字段模型
- 明确控制区、耦合、同步策略和 fallback 策略的配置承载方式

### D2. 明确配置优先级

建议方向：

- 明确 `env.properties`、`agent.properties`、远程 YAML 的职责边界
- 将 DMPC 专项配置优先放入远程 YAML

### D3. 补齐验证体系

建议方向：

- 建立协议、运行时、Agent、DMPC 扩展四类验证覆盖面
- 具体测试实现由工程侧承担，本文只规定验证主线

### D4. 补齐文档体系

建议方向：

- 保持架构说明、领域模型、验证策略、DMPC 演进方案和任务拆解口径一致
- 对接指南、快速接入、脚手架说明和交付包说明交由下游角色承接

### D5. 新增中央对接样例方向

建议方向：

- 提供接收中央目标、执行反馈回传和边界量更新的联调方向
- 将其作为中央与 Python 局部控制器对接的验证样板

## 9. 模块级架构映射

### 协议模块

涉及路径：

- `hydros_agent_sdk/protocol/`

方向：

- 拆分核心协议与 DMPC 扩展协议
- 保持统一分发入口和兼容导出层

### 运行时模块

涉及路径：

- `coordination_client.py`
- `coordination_callback.py`
- `state_manager.py`
- `message_filter.py`
- `multi_agent.py`

方向：

- 接入新命令
- 新增控制区运行时状态
- 增强多 Agent 路由
- 补齐日志上下文和过滤规则

### Agent 模块

涉及路径：

- `hydros_agent_sdk/agents/`

方向：

- 整理现有 Agent 抽象层职责

### 配置模块

- 增加 DMPC 配置样例方向

## 10. 里程碑建议

### M1：对象和协议层完成

完成标准：

- DMPC 对象和命令具备统一建模口径
- 新旧协议兼容路线明确
- 基础验证主线成立

### M2：运行时增强完成

完成标准：

- 状态管理和多 Agent 宿主可承载控制区场景
- 新命令收发链路成立

### M3：单区局部控制器样例完成

完成标准：

- 单区局部控制器最小闭环跑通
- 有对应配置、样例和验证说明

### M4：中央联调样例完成

完成标准：

- 中央目标下发到 Python 局部控制器链路跑通
- 执行反馈回传链路跑通

### M5：文档和验收收敛

完成标准：

- 文档口径完备
- 验收策略可执行
- 可以进入更复杂场景试点

## 11. 角色分工

在任务推进上，职责划分如下：

- `architect_agent`：定义阶段主线、架构映射、里程碑和放行口径
- `engineering_agent`：将本拆解转化为实施任务单、代码改造和验证记录
- `execution_agent`：承接样例落盘、联调材料、脚手架和交付归档

## 12. 风险控制要求

实施时必须注意：

- 不要把 DMPC 全局协调逻辑塞进 SDK 基础层
- 不要让 `BaseHydroAgent` 变成控制专用巨类
- 不要在没有对象模型的情况下先写复杂业务逻辑
- 不要跳过单区样例直接上多区协同
- 所有新协议对象都应先完成最小验证再进入联调

## 13. 结论

这份拆解清单的核心思想是：

- 先把 Python SDK 从“可运行的开放接入框架”升级为“可表达 DMPC 的开放接入框架”
- 再逐步让它承载局部控制器和支持性 Agent
- 最后通过中央对接样例把它纳入 `hydros` 的 DMPC 闭环体系

如果严格按此主线推进，Python SDK 的演进会保持边界清晰、改造可回退、验证可落地、文档可追踪。
