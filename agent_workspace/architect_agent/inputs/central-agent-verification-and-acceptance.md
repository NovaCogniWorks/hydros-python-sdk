# 中央智能体验证与验收方案

## 1. 文档目的

本文定义中央智能体 `hydros-agent-central` 的验证目标、验证分层、场景矩阵、验收指标与放行规则，用于支撑：

- 当前中央集中式调度能力验收
- 后续 DMPC 演进验证基线
- 场景化回归测试与异常测试

参考文档：

- `05-DMPC场景验证矩阵与验收标准.md`
- `03-场景如何运行.md`
- `SimCoordinationCallbackImpl.java`
- `CentralSchedulingAgent.java`
- `MpcControllerManager.java`

## 2. 验证目标

中央智能体验证不应只关注“是否调用了 MPC”，而应验证五个问题：

- 场景是否能稳定初始化
- 协同协议链路是否闭环
- 中央调度状态是否一致
- 优化结果是否能正确落地并下发
- 异常情况下是否能明确失败或退化

## 3. 验证原则

建议坚持以下原则：

- 先验初始化，再验控制质量
- 先验单区，再验多区
- 先验 `NORMAL`，再逐级扩展 `SIL/HIL/XIL`
- 先验闭环存在，再验闭环性能
- 先验异常可解释，再验性能优劣

## 4. 验证分层

### 4.1 L0 代码级验证

目标：
验证中央核心类在单元级别上逻辑正确。

覆盖点：

- `ContextManager.create/getContext/getAgentByObjectId`
- `HydroModelContext.onAgentInstanceSiblingCreated/getObject/getUpstreamObject`
- `MpcTaskState.activeNewRolling/registerHydroEvent`
- `CentralSchedulingAgent.getRollSteps/getMpcConfigUrl`
- `MpcClient` 的时序插值逻辑
- `MpcControllerManager.sendOpeningData` 的设备类型路由逻辑

### 4.2 L1 组件级验证

目标：
验证中央服务内部组件协作正确。

覆盖点：

- `SimTaskInitRequest -> ContextManager.create`
- `TimeSeriesDataUpdateRequest -> CentralSchedulingAgent.onTimeSeriesChanged`
- `TickCmdRequest -> CentralSchedulingAgent.onTickReceived`
- `MpcTaskState -> MpcClient -> MpcControllerManager`
- `MpcOptimizeResponse -> saveToData -> publishCommandEvent`

### 4.3 L2 协议级验证

目标：
验证中央节点作为协同协议参与者行为正确。

覆盖点：

- 初始化命令处理成功
- Tick ACK 返回完整
- 时序更新响应完整
- 兄弟 Agent 实例状态同步完整
- 失败响应的错误码和错误信息完整

### 4.4 L3 场景级验证

目标：
验证中央智能体在真实场景下闭环成立。

覆盖点：

- 场景配置可装配
- 对象模型可解析
- 对象归属可建立
- MPC 可触发
- 动作可下发
- 结果可追溯

### 4.5 L4 演进级验证

目标：
验证当前中央架构是否适合作为 DMPC 演进底座。

覆盖点：

- 是否可抽象出控制区
- 是否可定义边界量
- 是否可支持局部重算口径
- 是否可增加退化控制状态

## 5. 验证场景矩阵

### 5.1 单区基础场景

目标：
验证中央调度单区闭环成立。

验证内容：

- 中央初始化成功
- 首次时序事件能触发 `MpcTaskState`
- Tick 到来时按滚动步长触发重算
- 求解结果能保存
- 第一时域动作能下发

通过标准：

- 所有链路成功率 `100%`
- 无空上下文、无空归属、无丢 ACK

### 5.2 单区异常场景

目标：
验证中央对异常输入有明确响应。

验证内容：

- 空时序事件
- 时序步长不连续
- 时序全过期
- 传感器缓存为空
- MPC 返回失败
- `saveToData` 失败

通过标准：

- 必须有明确错误日志和失败响应
- 不允许静默失败
- 不允许中央状态不可恢复地破坏

### 5.3 双区耦合准备场景

目标：
验证中央是否具备未来双区 DMPC 改造准备度。

验证内容：

- 两组对象归属不同 Agent
- 中央可以同时寻址两个目标 Agent
- 上下游关系查询正确
- 两侧事件进入统一调度缓冲

通过标准：

- 两区归属映射无冲突
- 路由下发无错发

### 5.4 多事件组合场景

目标：
验证多类扰动在中央的统一调度入口是否稳定。

验证内容：

- 天气边界事件
- 用水需求事件
- 故障事件
- 多类事件叠加

通过标准：

- 都能进入中央时序处理逻辑
- 故障事件能形成固定控制约束
- 用水事件能形成目标输入

### 5.5 协同协议场景

目标：
验证中央节点对协调器协议的遵循性。

验证内容：

- `onSimTaskInit`
- `onTick`
- `onTimeSeriesDataUpdate`
- `onAgentInstanceSiblingCreated`
- `onAgentInstanceSiblingStatusUpdated`

通过标准：

- 请求必有响应
- 响应状态与错误码一致
- ACK 不丢失

## 6. 关键测试用例

### 6.1 初始化类

- 场景配置 URL 正确时，中央上下文创建成功
- 对象建模 URL 错误时，抛出业务异常
- 兄弟 Agent 返回 `managedTopObjects` 后，对象归属映射建立成功
- 同一 `bizSceneInstanceId` 二次初始化时，行为需可定义并可追踪

### 6.2 调度触发类

- 首次时序事件触发 `MpcTaskState` 创建
- 重复时序事件仅追加事件，不重复创建任务
- Tick 到达非滚动步时不触发重算
- Tick 到达滚动步时触发重算

### 6.3 输入构造类

- 时序正好命中当前步时，边界值收集正确
- 时序跨步插值时，插值结果正确
- 设备故障事件转固定控制约束正确
- 用水事件转目标序列正确

### 6.4 结果执行类

- `Gate` 类型设备转 `HydroDirectGateOpeningRequest`
- 非 `Gate` 类型设备转 `DisturbanceNodeWaterFlowRequest`
- 找不到对象归属 Agent 时，命令下发失败且日志明确
- 最优方案筛选逻辑正确

### 6.5 异常退化类

- MPC 服务异常时抛出明确业务异常
- 传感器缓存为空时触发重试逻辑
- `saveToData` 失败不应阻断当前进程崩溃
- 命令发布失败应记录错误上下文

## 7. 验收指标

### 7.1 强制指标

以下指标建议作为一票否决项。

- 场景初始化成功率 `100%`
- 上下文创建成功率 `100%`
- 对象归属映射成功率 `100%`
- Tick ACK 完整率 `100%`
- 时序事件处理成功率 `100%`
- 控制命令寻址正确率 `100%`
- 调度结果追溯完整率 `100%`

### 7.2 运行观察指标

以下指标作为优化观察项。

- 中央初始化耗时
- 首次调度耗时
- 滚动重算平均耗时
- MPC RPC 成功率
- 控制命令发布成功率
- 传感器缓存命中率
- 时序插值触发比例

### 7.3 异常治理指标

- 异常场景日志完整率 `100%`
- 失败响应错误码完整率 `100%`
- 无静默失败
- 无状态污染导致的持续性失控

## 8. 验收证据链

每次场景验收建议保留以下证据：

- 场景配置快照
- 任务实例 `bizSceneInstanceId`
- 中央上下文创建日志
- 兄弟 Agent 归属注册结果
- 时序事件输入记录
- MPC 请求与响应记录
- MPC 保存结果记录
- 控制命令发布记录
- Tick/Ack 记录
- 异常日志与失败响应记录

建议按任务实例归档：

```text
验收批次
  -> 场景配置
  -> 任务实例ID
  -> 初始化记录
  -> 时序输入记录
  -> 求解请求响应
  -> 控制下发记录
  -> ACK记录
  -> 异常记录
  -> 验收结论
```

## 9. 当前测试缺口

从现有工程看，主要缺口有四类：

- 单元测试不足
- 组件联调测试不足
- 异常与退化测试缺失
- 场景级自动化验收缺失

现有测试文件不足以支撑正式验收，只能算手工联调起点。

## 10. 验收结论模板

建议中央智能体验收结论固定采用以下结构：

- 场景信息
  场景 ID、任务实例 ID、环境类型、Agent 组合
- 通过项
  初始化、协议、调度、下发、追踪
- 失败项
  具体失败步骤、错误码、影响范围
- 风险项
  并发、事件积累、缓存依赖、外部服务依赖
- 结论
  `通过 / 限制通过 / 不通过`

## 11. 总结

这套验证与验收方案的核心不是证明“中央智能体算法先进”，而是证明：

- 它能被正确初始化
- 它能正确接入 `hydros` 协同链路
- 它能稳定维护中央调度状态
- 它能把求解结果正确下发到执行侧
- 它在异常条件下可解释、可追踪、可治理
