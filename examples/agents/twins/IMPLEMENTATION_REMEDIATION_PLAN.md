# Twins Agent 上线整改方案

本文档面向 `examples/agents/twins` 当前实现的生产化改造，目标不是重复说明代码结构，而是给出一份可执行的整改方案，覆盖问题归类、改造目标、设计建议、实施步骤、验收标准和交付顺序。

适用范围：

- `examples/agents/twins/twins_agent.py`
- `examples/agents/twins/hydraulic_solver.py`
- `examples/agents/twins/simulation_states.py`
- `hydros_agent_sdk/agents/tickable_agent.py`
- `hydros_agent_sdk/agents/twins_simulation_agent.py`

---

## 1. 改造目标

当前 `twins` 示例已经具备“可运行的 SDK 接入样板”，但还不具备“可稳定上线的数字孪生仿真 Agent”要求。上线整改的总体目标如下：

1. 保证边界条件能够被正确接收、缓存、映射并传入求解器。
2. 保证平台拓扑、求解器模型、输出对象三者之间的对象关系一致。
3. 保证求解参数、设备控制量和模型配置可以被显式注入，而不是停留在 demo 默认值。
4. 保证输出指标完整、命名统一、元数据可追踪。
5. 保证任务级资源隔离、异常诊断、失败回收和验证机制可满足生产运行要求。

---

## 2. 当前问题分级

为了便于安排优先级，建议把问题分成 P0、P1、P2 三类。

### 2.1 P0：阻断上线的问题

这类问题不解决，服务即使能跑，也不能保证结果可信。

#### P0-1 边界条件字段名不统一

当前在 `twins_agent.py` 中收集的默认边界条件指标名是：

- `inflow`
- `upstream_water_level`

而 `hydraulic_solver.py` 中更新边界时识别的字段是：

- `h_i_t`
- `Inflow_i_t`
- `qtot_i_t`

影响如下：

- 外部时序消息可能已经到达缓存，但求解器完全未使用
- 平台侧难以察觉，因为不会必然报错
- 最终表现为仿真结果不受真实边界驱动

整改要求：

- 建立统一的边界条件字段规范
- 在 Agent 层增加显式映射，不依赖调用方自己猜字段名
- 对未识别字段输出明确警告

#### P0-2 `boundary_params` 初始化缺失

`HydraulicSolver.solve_step()` 只会在 `self.boundary_params` 已存在对象时更新边界，但当前初始化阶段没有构造该结构。

影响如下：

- 所谓“实时边界驱动仿真”可能根本未生效
- 代码流程完整，但业务行为缺失

整改要求：

- 在 `HydraulicSolver.initialize()` 中根据模型或拓扑建立边界节点索引
- 明确上游边界、下游边界、边界对象 ID 与求解器节点 ID 的映射
- 增加初始化完成后的自检日志

#### P0-3 拓扑与求解器模型缺少一致性校验

当前上层拓扑来自 `hydros_objects_modeling_url`，底层求解器模型来自 `idz_config_url` 下载后的 YAML。两者之间没有一致性校验。

影响如下：

- 输出结果可能映射到错误对象
- 边界条件可能喂给错误节点
- 断面对象与管道对象关系可能与求解器内部不一致

整改要求：

- 引入对象映射校验层
- 在任务初始化时输出“拓扑对象数、模型对象数、匹配数、未匹配数”
- 对关键对象类型执行强校验：`Pipe`、`GateStation`、`DisturbanceNode`、`CrossSection`

### 2.2 P1：严重影响可用性的问题

这类问题不一定阻断运行，但会明显影响结果质量和后续维护。

#### P1-1 求解参数未真正生效

当前 `time_step`、`convergence_tolerance`、`max_iterations` 会被读取，但没有实际注入 solver 或底层引擎。

整改要求：

- 设计 `HydraulicSolverConfig` 配置对象
- 在初始化时显式注入 solver
- 将生效值打印到启动日志中，而不是只打印读取值

#### P1-2 设备控制量是写死的 demo 值

当前每个设备默认控制量固定为：

- `e_i_t=65.0`
- `n_i_t=1`

整改要求：

- 支持从配置文件或远程任务参数加载设备初值
- 支持后续控制量更新
- 加入控制量合法性检查，例如开度范围、机组数量范围

#### P1-3 Pipe/GateStation 断面信息映射不完整

当前 `cross_section_info` 没有被真正填充，因此断面名称和父子关系不完整。

整改要求：

- 基于拓扑完整构造断面索引
- 输出断面对象真实名称
- 明确首断面和末断面的流量定义

#### P1-4 输出指标过于简化

当前只输出 `water_level` 和 `water_flow`，不足以支持较复杂的业务场景。

整改要求：

- 梳理场景需要的最小指标集
- 按对象类型输出必要指标
- 明确哪些指标属于节点级、设备级、断面级

### 2.3 P2：影响维护性与稳定性的问题

#### P2-1 代码中存在编码乱码注释

这不影响运行，但会直接降低后续维护效率。

整改要求：

- 统一文件编码为 UTF-8
- 清理乱码中文注释
- 补充关键逻辑块注释

#### P2-2 初始化、自检、清理日志还不够结构化

整改要求：

- 统一关键日志格式
- 在日志中输出任务 ID、对象数量、边界数量、指标数量
- 区分告警类日志与功能失败类日志

#### P2-3 缺少系统化测试与验收脚本

整改要求：

- 增加最小集成验证
- 增加边界条件映射测试
- 增加初始化一致性校验测试

## 3. 目标架构建议

建议将当前实现从“示例拼接结构”调整为“可上线的四层结构”。

### 3.1 推荐分层

1. 输入适配层  
   负责接收平台时序消息、标准化边界条件字段、做 step 对齐和缺值处理。

2. 模型映射层  
   负责维护平台对象 ID、拓扑对象、求解器节点之间的映射关系。

3. 求解执行层  
   负责初始化求解器、注入控制量、执行单步仿真、回收资源。

4. 输出发布层  
   负责将求解结果转换为平台统一指标结构并发布 MQTT。

### 3.2 推荐新增的数据对象

建议增加以下配置与映射对象：

- `BoundaryConditionMapping`
- `SolverObjectMapping`
- `HydraulicSolverConfig`
- `MetricsPublishingPolicy`

建议作用如下：

- `BoundaryConditionMapping`：统一外部指标名与 solver 内部字段名
- `SolverObjectMapping`：维护拓扑对象和求解器节点的双向映射
- `HydraulicSolverConfig`：统一管理求解参数与默认值
- `MetricsPublishingPolicy`：按对象类型约束输出指标策略

## 4. 具体整改方案

以下方案按模块展开。

## 4.1 `twins_agent.py` 改造方案

### 4.1.1 增加边界条件标准化逻辑

在 `_collect_boundary_conditions()` 前增加一层标准化方法，例如：

- `_normalize_boundary_metric_code(metrics_code)`
- `_build_boundary_conditions_for_step(step)`

目标：

- 把平台侧输入统一映射为 solver 可识别字段
- 支持别名，例如：
  - `inflow` -> `Inflow_i_t`
  - `upstream_water_level` -> `h_i_t`
  - `flow` -> `qtot_i_t`

建议行为：

- 未识别字段输出 `warning`
- 冲突字段按优先级保留
- 每个 step 输出“采集到的边界对象数”和“实际生效字段数”

### 4.1.2 增加初始化一致性检查

在 `_initialize_twins_model()` 结束前增加自检：

- 拓扑对象数量
- Pipe 数量
- GateStation 数量
- DisturbanceNode 数量
- 断面对象数量
- solver 内部节点数量
- 成功映射数量
- 未映射对象列表

如果核心对象存在大面积未映射，应直接中断初始化。

### 4.1.3 强化指标输出策略

对 `_convert_results_to_metrics()` 进行重构：

- 将 `node_info` 构造逻辑拆成独立方法
- 完整填充 `cross_section_info`
- 明确每类对象输出哪些指标
- 将硬编码规则抽到策略对象或配置中

建议输出策略至少区分：

- 节点类对象
- 干线对象
- 断面对象
- 设备对象

### 4.1.4 强化 `on_boundary_condition_update()`

当前方法只是记录日志和更新本地状态。建议增加：

- 边界条件计数统计
- 未识别对象统计
- 超前 step / 过期 step 统计
- 重复数据覆盖统计

这层不一定直接更新 solver，但至少要让问题可观测。

## 4.2 `hydraulic_solver.py` 改造方案

### 4.2.1 显式引入配置对象

建议给 `HydraulicSolver` 增加配置对象，例如：

```python
@dataclass
class HydraulicSolverConfig:
    time_step: int = 60
    convergence_tolerance: float = 1e-6
    max_iterations: int = 100
```

然后在 `initialize()` 时接收该配置，避免求解参数只是打印不生效。

### 4.2.2 初始化阶段构造边界参数索引

整改要求：

- 明确哪些对象是边界对象
- 构造 `self.boundary_params`
- 对每个边界对象建立：
  - `upstream_boundary`
  - `downstream_boundary`
  - solver node id
  - platform object id

如果当前底层 `HydroSimulator` 可以提供边界节点信息，应优先使用其原生接口；如果不能，则从拓扑和 YAML 配置中做解析。

### 4.2.3 初始化阶段构造对象映射

建议新增内部结构：

```python
self.object_mapping = {
    platform_object_id: {
        "solver_node_id": ...,
        "object_type": ...,
        "object_name": ...,
    }
}
```

用途如下：

- 输入时边界条件可以正确找到 solver 节点
- 输出时结果可以映射回平台对象
- 自检时可打印映射覆盖率

### 4.2.4 扩展单步求解输出

当前只输出：

- `water_level`
- `water_flow`

建议扩展为“分层输出”：

- 节点级指标：`water_level`、`water_flow`
- 设备级指标：`gate_opening`、`pump_status`、`efficiency`
- 边界级指标：`inflow`、`outflow`
- 可选诊断指标：收敛状态、迭代次数

即使第一阶段不全部开放，也建议在 solver 内部保留结构。

### 4.2.5 增强失败诊断

在 `solve_step()` 异常分支中，建议输出：

- 当前任务 ID
- step
- 边界对象数量
- 控制量对象数量
- 是否完成初始化
- 最近一次成功 step

这类信息对生产问题定位非常关键。

## 4.3 `simulation_states.py` 改造方案

### 4.3.1 清理未使用或未接入的状态定义

建议分两步：

1. 保留生产真正会用到的状态对象
2. 对预留结构打上明确注释，说明当前是否已接入

目标：

- 降低维护人员理解成本
- 避免看起来“好像支持”，但实际没有被使用

### 4.3.2 统一字段命名风格

当前字段混有：

- 平台风格命名
- 数学符号风格命名
- 英文业务字段命名

建议做统一规则：

- 对外发布字段用业务语义名
- solver 内部字段可保留数学风格名
- 两者之间通过映射层转换

## 5. 验证方案

上线整改不能只改代码，必须配套验证。

## 5.1 单元测试

建议至少补以下测试：

### 5.1.1 边界字段映射测试

验证输入：

- `inflow`
- `upstream_water_level`
- `flow`

能够被正确映射成 solver 字段。

### 5.1.2 step 取值测试

验证：

- 当目标 step 存在时，取到对应值
- 当目标 step 不存在时，按预期返回空值或最近值

### 5.1.3 对象映射测试

验证：

- 平台对象 ID 可以找到 solver node ID
- 不存在映射时抛出清晰错误或告警

## 5.2 集成测试

建议构造最小案例：

1. 加载一套小规模拓扑
2. 加载对应 IDZ 配置
3. 注入边界条件
4. 执行至少两个连续 step
5. 验证结果指标有变化且与边界变化一致

重点检查：

- 初始化是否成功
- 边界是否生效
- 输出对象是否正确
- 任务结束时资源是否清理

## 5.3 回归测试

建议验证以下行为在整改后不回退：

- 多任务并发运行
- 不同任务独立回收
- 无边界输入时仍可执行默认步进
- 错误情况下服务不会崩溃退出

## 6. 验收标准

以下标准建议作为上线前验收门槛。

### 6.1 功能验收

1. 给定边界条件时，求解结果必须发生可解释变化。
2. 核心对象类型必须全部建立映射关系。
3. Pipe / GateStation 的断面输出必须具备真实名称和对象 ID。
4. 求解参数必须能够通过配置显式控制并在日志中确认生效值。

### 6.2 稳定性验收

1. 多任务并发运行时状态不得串线。
2. 任务终止后临时配置文件必须被清理。
3. solver 初始化失败时必须返回失败响应，不得进入半初始化状态。
4. 单步求解失败时服务进程不得崩溃。

### 6.3 可观测性验收

1. 日志必须能定位到任务 ID、step、对象数量、边界数量、指标数量。
2. 对未识别字段、未映射对象、缺失边界参数要有明确告警。
3. 初始化和终止阶段要有结构化摘要日志。

## 7. 推荐实施顺序

建议分三期实施，降低风险。

### 一期：修通核心数据链路

目标：

- 保证边界条件真的进 solver
- 保证对象映射是正确的

实施项：

1. 统一边界字段映射
2. 构造 `boundary_params`
3. 建立对象映射表
4. 加入初始化一致性自检

上线标准：

- 仿真结果可随边界变化而变化

### 二期：补齐参数和输出

目标：

- 让仿真行为可控、结果可用

实施项：

1. 引入 `HydraulicSolverConfig`
2. 支持控制量初始化和更新
3. 完整填充断面信息
4. 扩展输出指标

上线标准：

- 输出可以满足基础业务消费

### 三期：完善验证与运维

目标：

- 提升可维护性和可观测性

实施项：

1. 增加单元测试和集成测试
2. 统一日志规范
3. 清理乱码注释和无效结构
4. 增加故障诊断信息

上线标准：

- 满足持续运行和问题定位要求

---

## 8. 最终建议

这套 `twins` 示例不建议直接作为生产实现上线，而应作为“生产实现的骨架”。整改时应坚持两个原则：

1. 不要继续把业务规则散落在 `twins_agent.py` 的多个方法中。应尽快收敛成显式的映射层、配置层和输出策略层。
2. 不要只关注“能跑起来”。数字孪生场景里最危险的问题不是进程挂掉，而是静默地产出错误结果。所有关键输入和映射关系都必须可验证、可观测。

如果要继续推进，建议下一步直接产出一份“函数级改造清单”，把每个整改项落实到具体类、方法和新增对象上，便于开发排期和实施。
